WindowLayouter

Monitor-aware window layout manager — Help & Reference

📐Overview

WindowLayouter is a lightweight Windows utility that arranges open application windows into predefined tile layouts on a target monitor. Layouts are described as presets: each preset encodes the grid shape, which windows to include, and on which monitor to apply them. Presets can be triggered from the toolbar, the system tray, or a global hotkey.

Key capabilities:

  • Grid tiling — divide the monitor work area into an N×M grid and fill it with selected windows.
  • Split tiling — separate windows into a tiled left column and a right stack by process/title pattern.
  • Per-preset activation guards — restrict a preset to a specific monitor resolution class or device.
  • Global hotkeys — apply any preset without switching focus.
  • Live language switching — UI labels are loaded from external .ini files at runtime.
  • Single-instance design — a second launch brings the existing window to the front.

🚀Quick start

  1. Launch WindowLayouter.exe. The app starts in the main window and adds a tray icon.
  2. Select one or more rows in the window list.
  3. Click a preset button in the toolbar (e.g. 4K > 2×2).
  4. The selected windows are rearranged on the target monitor instantly.
Tip If the preset buttons are greyed out, the required monitor is not connected. See Activation conditions for details.

To customise hotkeys, patterns or grid dimensions, open the Preset Editor from the toolbar or the tray menu. Settings are saved automatically to settings.ini.

🗂️Window list

The list on the left side of the main window shows all discovered top-level application windows. Each row represents one window with its title, process name, window class, monitor device name, and HWND (hexadecimal handle).

Selection

Click a row to select it. Hold Ctrl to add rows, hold Shift to extend the range. There are no checkboxes — standard list-view multi-select applies. The selection determines which windows a preset with Selected rows scope will act on.

Sorting

Click any column header to sort by that column. Click again to reverse the order. Sorting is alphabetical for text columns and numeric for the HWND column.

Filtering

Use the filter fields above the list to narrow down which windows are shown. Filtering does not change the full window set — it only controls what is visible and therefore what All visible rows scope will act on.

Excluded windows

The following windows are never shown in the list:

  • Windows without a title.
  • Tool windows (WS_EX_TOOLWINDOW) and no-activate windows.
  • Cloaked windows (e.g. UWP apps on inactive virtual desktops).
  • Very small windows — narrower than 60 px or shorter than 40 px.
  • Shell internals: Progman, WorkerW, Shell_TrayWnd.

🔍Filters

Two filter fields sit above the window list: Title and Process. Both support wildcard patterns. Filtering is live — the list updates on every keystroke.

FieldMatches againstExample
TitleWindow title text*Messenger*
ProcessExecutable file name (basename)*chrome*

Leaving a filter field empty means match everything — no filtering on that field. Both filters are applied simultaneously (AND logic).

Tip Combine both filters to isolate a specific window precisely. For example, Title: *Meeting* + Process: *teams* shows only Teams windows whose title contains "Meeting".

📑Tabs

The right panel of the main window contains three tabs.

Monitors

Displays a text dump of every monitor detected by the system, including device name (e.g. \\.\DISPLAY1), pixel bounds, work area, resolution class, DPI and whether it is the primary display. The path to settings.ini is shown at the top for quick reference.

Presets

Shows a summary of all defined presets: their tray label, toolbar label, selection scope, activation status and layout description. Use this tab to review the current preset configuration at a glance. To edit presets, click Open Preset Editor.

Rules

Contains two controls and a reference text panel:

  • App Selection Override — a global override that replaces every preset's individual Selection scope with one fixed mode. Set to Follow preset to let each preset use its own setting.
  • Language — switch the UI language. See Language.

The text area below explains selection, filtering, layout, activation, spacing and ordering rules — it mirrors the content of this help file in condensed form.

🎛️What is a preset?

A preset is a named layout recipe. When you apply a preset, WindowLayouter:

  1. Refreshes the window and monitor lists.
  2. Verifies that the preset's activation conditions are met.
  3. Collects candidate windows according to the selection scope.
  4. Arranges those windows using the preset's Grid or Split layout.

Presets are saved to settings.ini and survive application restarts. The built-in presets can be fully customised; their count is fixed at five, but all fields — name, hotkey, grid dimensions, patterns and activation guards — are editable.

Window ordering within a layout follows the system z-order (front-to-back) after pattern filtering — the frontmost matching window goes into tile 0.

▦Grid mode

Grid mode divides the target monitor's work area (excluding the taskbar) into a uniform N×M tile grid and places one window per tile in row-major order (left to right, top to bottom).

1
3
2
4

Example: 2×2 grid with four windows.

If the number of selected windows is smaller than the total cell count, only the first N cells are used and the rest are left empty. If there are more windows than cells, the excess windows are skipped (reported as skipped in the status bar).

A gap of window_gap_px pixels is applied around each tile (configurable in settings.ini). Set it to 0 for seamless tiling.

Grid columns and rows are set in the Preset Editor. The Right Stack Width % field is disabled for Grid mode and has no effect.

◫Split mode

Split mode divides the monitor into a left column (a tile grid) and a right column (a single stacked area). Each candidate window is routed to one side based on pattern matching.

Left 1
Left 2
Left 3
Left 4
Right

Example: Split 2×2 with 4 left tiles and 1 right stack.

Routing logic

  • Windows matching Left Title or Left Process patterns are placed in the left column grid (N×M tiles).
  • Windows matching Right Title or Right Process patterns are all stacked on top of each other in the right column (last one on top).
  • A window can match both sides — it will appear in both if it matches both patterns. Empty patterns match all windows on that side.

Right stack width

The Right Stack Width % field controls what percentage of the monitor width is given to the right column. Valid range: 20–80%. The left column takes the remainder. For example, a value of 33 gives the right column 1/3 of the monitor and the left column 2/3.

Tip The layout preview in the Preset Editor updates live as you change fields, showing labels derived from your own patterns.

🎯Selection scope

The selection scope controls which windows a preset treats as its input candidates:

ScopeDescription
Selected rows Only the rows currently selected (highlighted) in the window list. Default for most presets — gives precise manual control.
All visible rows Every row currently visible in the list after filter application. Useful for quickly tiling a filtered subset without selecting individually.
All discovered windows Every window that passed the initial enumeration, ignoring the filter and selection. Broadest scope — tiles everything WindowLayouter can find.

App Selection Override

The App Selection Override dropdown in the Rules tab supersedes every preset's individual scope setting. Set it to Follow preset (the default) to let each preset use its own scope. Change it to one of the three modes to force all presets to use the same scope globally, regardless of what the Preset Editor says.

🔒Activation conditions

A preset can be restricted to specific monitor configurations. If the conditions are not met, the preset button and tray menu item are greyed out, and applying the preset does nothing except display an explanatory status message.

ConditionDescription
Require external monitor The preset is only active when at least one non-primary monitor is connected. Unchecking this allows the preset to target the primary display.
Resolution class Restricts the preset to monitors of a specific resolution class. See Resolution classes. Set to Any to ignore resolution.
Monitor device An optional wildcard matched against the monitor device name (e.g. \\.\DISPLAY2). Leave blank for any device. The first monitor that satisfies all conditions becomes the target.

When all three conditions are set to their permissive defaults (external unchecked, class = Any, device empty), the preset falls back to the primary monitor and is always active.

Note All three conditions must be satisfied simultaneously. If the first monitor in the system list doesn't match, the next one is tried, and so on.

⌨️Hotkeys

Each preset can have a global hotkey that applies it from any application, even when WindowLayouter is minimised to the tray.

Format

Hotkeys use the format: Modifier+…+Key

ComponentAccepted values
ModifierCtrl, Control, Alt, Shift, Win, Windows
KeySingle letter A–Z, digit 0–9, or function key F1–F24

Examples

Ctrl+Alt+1
Ctrl+Alt+F2
Win+Shift+T

Hotkeys are registered at startup and re-registered each time you save a preset. If registration fails (another app owns the same combination), the hotkey is silently disabled for that preset.

To remove a hotkey, clear the Hotkey field in the Preset Editor and save.

Note At least one modifier key is implicitly required by Windows — bare keys like F2 alone cannot be registered as global hotkeys.

💾Save vs Apply

The Preset Editor has two action buttons:

ButtonWhat it does
Save Preset Validates and writes the current form values to g_presets in memory and persists them to settings.ini. Updates the preset list, toolbar buttons and hotkey registrations. Does not rearrange any windows.
Apply Preset Saves first (same as Save Preset), then immediately applies the preset — rearranging windows on the target monitor. Equivalent to saving and then clicking the toolbar button.
Tip Use Apply Preset during setup to see the layout live while you tweak columns, rows and width without leaving the editor.

✳️Wildcard patterns

Filter fields, Left/Right process and title fields, and the Monitor Device field all support two wildcard characters:

CharacterMeaning
*Match any sequence of characters (including none).
?Match exactly one character.

Matching is case-insensitive.

Examples

PatternMatches
*chrome*Any string containing "chrome" anywhere.
chrome.exeExactly "chrome.exe" (no wildcards).
*Antigravity*Any window title containing "Antigravity".
report_202?.pdfe.g. "report_2024.pdf" but not "report_20244.pdf".
Performance note Patterns are limited to a maximum of 2 asterisks to prevent exponential backtracking. Patterns with 3 or more * characters will never match anything.

🖥️Resolution classes

Resolution classes are broad monitor categories used as activation conditions. They classify a monitor by its physical pixel dimensions, handling portrait-rotated monitors correctly by using the larger dimension as "width" for classification.

ClassApproximate resolutionPixel threshold
AnyNo restriction—
2K2560×1440 (QHD)≥ 2500×1400 (landscape)
2.5K2560×1600≥ 2500×1550 (landscape)
4K3840×2160 (UHD)≥ 3800×2100 (landscape)

Thresholds are intentionally generous — the goal is broad category matching, not exact panel identification. A 2560×1600 panel classifies as 2.5K, not as 2K, because 1600 > 1550.

Tip You can see a monitor's detected resolution class in the Monitors tab.

📥System tray

WindowLayouter places an icon in the Windows notification area (system tray).

ActionResult
Double-click iconShow main window and bring it to the foreground.
Right-click iconOpen context menu.

Context menu items

  • Show — bring main window to the foreground.
  • Preset Editor — open the Preset Editor window.
  • About — open the About dialog.
  • Hide to Tray — hide the main window (keeps running in background).
  • Preset shortcuts — one entry per preset with Show in tray enabled, with its hotkey shown in brackets. Greyed-out when the required monitor is absent.
  • Exit — fully quit WindowLayouter.
Note Closing the main window with the × button hides it to the tray rather than exiting. Use Exit in the tray menu to terminate the process.

📄Settings file

All settings are persisted to a plain-text INI file: 

%LOCALAPPDATA%\WindowLayouter.Native\settings.ini

The file is created automatically on first launch with default values. It is saved automatically whenever a setting changes (language switch, scope override, preset editor save, window close).

File format

Standard INI format: [Section] headers, key=value pairs. Inline comments start with // and are stripped when the file is read. 

[Window]
left=100
top=100
width=1760
height=1040
tab=1                  // Active tab: 0=Monitors, 1=Presets, 2=Rules
window_gap_px=4        // Gap between tiles in pixels

[App]
language=en            // Language code — file must exist in lang\
selection_scope_override=Follow preset

[Hotkeys]
PresetGrid2x2=Ctrl+Alt+1
PresetSplit2x2=Ctrl+Alt+2
...

[Preset0]
name=4K > 2x2
kind=Grid
columns=2
rows=2
require_external_monitor=1
required_resolution_class=4K
...
Note You can edit settings.ini directly in a text editor. The app reloads language files automatically when they change on disk (poll interval: 1.5 s), but settings.ini changes require a restart to take effect.

🌐Language

The UI language is loaded at runtime from a file in the lang\ folder next to WindowLayouter.exe: 

lang\en.ini    ← English (default)
lang\pl.ini    ← Polish

Switch language using the Language dropdown in the Rules tab. The change takes effect immediately across all windows including the Preset Editor. Language files are monitored for on-disk changes — editing a file is reflected live without restarting.

Adding a language

Copy lang\en.ini, name it lang\code.ini (e.g. lang\de.ini) and translate the values. The file must be saved as UTF-8 (with or without BOM). Restart the app and the new language will be selectable from the dropdown.

Note If a selected language file cannot be found, the app silently falls back to lang\en.ini. If that file is also missing, built-in English fallback strings are used.
